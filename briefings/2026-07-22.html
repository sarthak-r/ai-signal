Voices

The team behind the leading coding agent says the quiet part: trust the model more, constrain it less.

The Claude Code team on how they actually build itsimonwillison.net · 21 Jul

Simon Willison's writeup of an AI Engineer World's Fair fireside with Anthropic's Cat Wu and Thariq Shihipar is the densest app-layer craft post in the window. The load-bearing details: they cut Claude Code's system prompt ~80% for Fable/Opus 4.8, and removing examples helped most — the frontier model "was just more creative" than the examples were boxing it into. Tool design is deliberately low-cardinality — "every tool we add has a distinct function from every other," and specialised search tools were replaced with plain bash. They now write prompts thinking about "the ways in which it could be misinterpreted by a well-intentioned human." Features must clear an internal retention bar among Anthropic staff before public release, and human code review is dropped only in categories the automated reviewer handles perfectly, with every incident fed back into the eval set. Their internal Claude Tag reportedly lands 65% of product PRs.

Four transferable moves, none of them "use a better model." One: if your agent's prompt is a wall of examples and edge-case rules written against last year's model, delete most of it and re-measure — over-constraint now costs you the model's judgment. Two: collapse overlapping tools; ambiguity about which tool to call is a bigger failure source than any single tool's design. Three: write instructions for a well-intentioned reader, not an adversarial parser — absolute rules get misapplied. Four: the honest way to remove a human check is Anthropic's — prove with evals that automation is perfect in a specific category, retire review only there, and feed every miss back. That's the exact template a regulated advice shop needs to defend dropping a manual review step.

Claude Code v2.1.217 — a ceiling on the agent swarmgithub · 21 Jul

The headline change is orchestration governance: a cap on concurrent subagents (default 20, override via CLAUDE_CODE_MAX_CONCURRENT_SUBAGENTS), and subagents no longer spawn nested subagents by default (CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH to allow deeper trees). Also fixed: a memory leak where truncated MCP tool outputs retained the full result, Windows auto-update failures that could delete claude.exe, corporate mTLS/proxy/OAuth settings being ignored in Desktop, and --max-budget-usd not stopping background subagents.

As multi-agent fan-out becomes the default way to work, runaway spawning is the new footgun — cost and stability both blow up when agents recursively launch agents. Making non-nesting the default and a concurrency ceiling explicit is a vendor telling you the shape of the problem before you hit it. If you run parallel agents in your own product, copy the two knobs directly: a hard concurrency cap and a spawn-depth limit are the cheapest insurance against a single request quietly becoming a hundred. The budget-stop and MCP memory-leak fixes matter for anyone leaving agents running unattended.

This is eval/observability-as-product aimed at the coding-agent layer itself — treating "is this agent worth its token bill" as a first-class ops question rather than a gut feel. Two things to take even if you never touch CloudWatch. One: the metric that matters isn't tokens, it's tokens correlated to shipped output (PRs, commits) — instrument that ratio for any agent you run in production. Two: in a regulated shop, a per-team spend-and-outcome audit trail for agent usage is exactly what a reviewer will eventually ask for; the pattern here is a free reference for what to log before you're asked.
