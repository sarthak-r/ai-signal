Claude Code 2.1.195 — hook matchers now exact-match, mouse-click kill switch, plugin-consent and background-job fixesAnthropic · Jun 26

The release after 2.1.193. The one that can change behaviour: hook matchers containing hyphens now use exact matching instead of substring matching — if your settings.json relied on a hyphenated matcher firing on partial matches, those hooks may stop firing after you update. Also new: CLAUDE_CODE_DISABLE_MOUSE_CLICKS to turn off mouse interaction in fullscreen (wheel scroll preserved), a fix so external plugins enabled via project settings no longer demand install consent on every loader path, a fix for background jobs losing data when written by a newer Claude Code version, and voice-dictation fixes (macOS silence capture; auto-submit for Japanese/Chinese/Thai).

Read the hook-matcher line before you upgrade a CI or agent box: a silent change in which hooks fire is exactly the kind of thing that breaks a guardrail or audit step without an error. Grep your hook configs for hyphenated matchers and confirm they still match what you expect. The plugin-consent and background-job fixes matter if you run Claude Code unattended in pipelines.

The latest Ollama release wires local models more cleanly into agent tooling: thinking-capability detection when launching against opencode, auto-install for Claude Code and opencode, better Codex model-drift detection, plus fixes for Windows hybrid-graphics GPU classification, MLX speculative decoding, CUDA sm_86 support, and generation headroom for shifted prompts.

If data residency or cost pushes you toward self-hosting a model behind your wealthtech stack, this shortens the wiring: one ollama launch now stands up a local or cloud model into Claude Code/opencode/Codex with the thinking flag detected for you. Worth a look if you've been keeping a local fallback tier or an on-prem option for sensitive client data.
