Understanding is the new bottleneck

Yesterday the story was self-verification — Willison's agents recording video proof their work runs, the World's Fair closing on Harness Engineering. Today Geoffrey Litt supplies the sequel and the sharper question. If agents now verify their own output, "did it run?" stops being the human's job — so the real bottleneck moves one level up: do I still understand what the agent built well enough to steer the next change? Litt calls understanding, not verification, the new constraint on iterative work, and ships concrete techniques — explainer docs gated by quizzes that act as a speed regulator on the agent loop. The rest of the day rhymes: Willison has an agent tune its own prompts with DSPy and publishes the coding loop as a tiny readable library; Claude Code's background agents now open their own draft PRs; MCP Apps put server-rendered UI inside the host; and Sonnet 5's new tokenizer quietly inflates your bill. The model is settled — now the fight is keeping a human in the loop who still knows what's going on.

Geoffrey Litt: "understanding is the new bottleneck" — and quizzes are how you throttle the agentGeoffrey Litt · 2 Jul

Litt (Ink & Switch) argues the point of understanding AI-written code was never mainly to verify it — agents increasingly self-verify — but to stay a real participant in an iterative project: without conceptual fluency you can't direct the next evolution, and "cognitive debt" compounds. His fixes are concrete: code explainer docs that teach intuition before showing a "literate diff" (prose-ordered, not alphabetical), micro-worlds you can step through like Papert's Mathland, and — the sharp one — embedded quizzes: "I won't send code to others until I can pass the quiz." The quiz is "a speed regulator" that stops the AI loop outpacing human comprehension. The frame: augment, not just automate.

This is the missing half of the self-verification story from yesterday. Wire proof-of-work into the agent loop, yes — but also wire proof-of-understanding into the human's. For a small team shipping fast, a comprehension gate (an explainer + quiz before merge) is a cheap governance primitive that keeps someone accountable actually able to reason about the system. In a compliance-bound domain, "the human understood it" is not optional — it's the audit trail.

Willison: point an agent at DSPy and let it tune its own SQL promptsSimon Willison · 2 Jul

Willison had Claude Fable 5 install DSPy against Datasette Agent and hunt for prompt improvements — evaluated with cheap GPT-4.1 mini/nano as the judges. It surfaced a real bug: the schema listing gave only table names, and prompt advice to avoid redundant describe_table calls pushed the agent to guess column names and spiral into error-retry loops. Fix: put column names in the schema listing, or soften the advice. The agent found and diagnosed its own prompt regression.

Prompt engineering becomes an eval-driven, agent-run loop rather than hand-tuning by vibes. The reusable move: give an agent your eval harness (DSPy or similar) plus a cheap judge model and let it propose prompt/context changes against a scored benchmark. The bug is also a lesson — "don't call the extra tool" advice that saves tokens can cost far more in retry loops. Measure the loop, not the single call.

SquidHub puts a whole team and their AI agents ("Squids") in shared real-time rooms — one shared context to brainstorm, plan and build, instead of everyone copy-pasting between private chats. The design choice worth stealing: turn-taking — one agent speaks per turn, so disagreement plays out as a conversation with trade-offs surfaced, not a wall of competing pop-ups, and teammates make the final call. Bring hosted Claude or your own Claude/GPT/Grok key.

"Multiplayer agents" is graduating from demo to product category. The interesting primitive isn't the shared room — it's turn-taking as a UX for disagreement: a clean answer to the "five agents, five contradictory answers" problem that kills group agent UX. If your product has multiple humans acting on shared AI output, watch how the reversible-vs-approval line and turn order get drawn here.

Build Patterns

Two primitives worth reading closely — one for UI, one for the loop itself.

MCP Apps: servers ship interactive UI, not just text — through the same consent path as a tool callMCP · spec

Beyond the stateless core and OAuth flagged yesterday, the 2026-07-28 spec adds MCP Apps: servers deliver server-rendered HTML in sandboxed iframes, using pre-declared UI templates the host can cache and security-review up front — and every UI-initiated action "goes through the same audit and consent path as a direct tool call." Paired with the Tasks extension, where a tools/call returns a task handle the client drives via tasks/get / update / cancel for long-running work.

This is generative UI with a governance model attached — a tool can hand back a real interface (a form, a chart, an approval widget) instead of a wall of text, and it inherits the host's permission and audit rails for free. For anything where a raw text agent response is too lossy — reviewing a plan, confirming a transaction, steering a projection — MCP Apps is the emerging surface. Design tools to return UI, not just JSON.

Willison's llm-coding-agent: the whole Claude Code–style loop in six tools you can readSimon Willison · 2 Jul

Willison released llm-coding-agent 0.1a0 — a minimal coding agent on his LLM library with just six tools (read/write files, run commands with a timeout, search, pattern-match). It was itself built agent-first: Claude wrote the spec, then implemented it via TDD across multiple commits, and the result can autonomously build things like a Swift ASCII-art CLI. The point isn't a Cursor rival — it's a legible reference for what the loop actually is.

If the harness is the product, you need to understand the harness (per Litt above) — and a readable six-tool implementation is the fastest way to internalise that a "coding agent" is a small tool set in a loop, not magic. Fork it to prototype a domain agent, or read it to demystify what your vendor's agent is doing under the hood before you build governance around it.

Your Radar

A cost signal hiding inside the model you just switched to.

Sonnet 5's new tokenizer inflates token counts up to 1.35× — your bill is not just the sticker priceCost · migration

Migrating to Sonnet 5 (now the Claude Code default at promotional $2/$10 per Mtok through 31 Aug) carries three breaking changes worth budgeting for: the new tokenizer generates 1.0–1.35× more tokens for the same text, adaptive thinking now defaults to "high" effort, and temperature/sampling parameters are removed. The headline per-token price dropped; effective cost per task may not.

Re-run your own cost benchmarks before trusting the $2/$10 sticker — a 35% token-count inflation plus thinking-on-by-default can erase the per-token savings on agentic workloads that already burn tokens in loops (see yesterday's Raschka finding that Claude Code uses ~2× the tokens of Codex). If you priced a feature or a customer tier on the old numbers, re-model it. And if you depended on temperature for output diversity, that lever is gone — rework it into prompting.
