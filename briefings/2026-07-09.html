Fix the process, not the code

The best build story of the last 48h isn't a model — it's a harness. Jarred Sumner rewrote the Bun runtime from Zig to Rust in 11 days with 64 concurrent Claude agents, and his own verdict on why a million lines of LLM-generated code were safe to merge names three things, none of them the model: a language-independent test suite with 1.38M assertions, blind reviewers told to "assume the code is wrong," and a rule to fix the process that generates the code, not the code itself. The rest of the day rhymes: Kenton Varda bans AI-written PR descriptions for stating what the code does while omitting why; Auto compiles the deterministic slices of an agent run into verified WASM so nothing is solved twice; and OpenAI's GPT-Live fronts a fast full-duplex voice and delegates the hard reasoning to a frontier model behind it. The through-line for a builder: the model is rented and interchangeable — the verification harness and the deterministic core are the assets you own.

Voices

Two practitioners on the same point from opposite ends: the value isn't the AI's output, it's the scaffold you build to trust — or reject — it.

Jarred Sumner ships Bun rewritten Zig→Rust by 64 agents — and credits the test suite, not the modelBun blog · 8 Jul

Bun's creator details the port: 11 days, 6,502 commits, ~1M lines, ~$165,000 in tokens (5.9B input / 690M output), run as Claude Code dynamic workflows with 64 concurrent agents across 4 git worktrees on pre-release Fable 5. What made it mergeable was verification, not generation: Bun's test suite is written in TypeScript, so it's runtime-independent — 1,386,826 expect() calls across 60,624 tests, zero skipped. Each implementer agent got two blind reviewer agents handed only the diff and told to "assume the code is wrong" — catching a use-after-free, a negative-timespec bug, and an eager-eval panic before merge. Sumner's summary: "a language-independent test suite with a million assertions, adversarial code review, and when something does go wrong, fixing the process that generates the code instead of hand-fixing the code."

This is the most complete public proof yet that agent-scale codegen is gated by your verification harness, not your model choice — the model was interchangeable; the million-assertion oracle and the blind adversarial reviewers were the moat. The transferable moves for a small team: (1) invest in tests that don't depend on the code under test so agents can't game them; (2) run review as a separate agent that only sees the diff and is told to break it; (3) when an agent produces bad code, patch the workflow, not the output — otherwise you're doing the work by hand. For a regulated stack this is the template for trusting generated change at all: the oracle is the control.

Kenton Varda declares a moratorium on AI-written PR and commit descriptionsWillison / X · 8 Jul

Cloudflare's principal engineer (Cap'n Proto, Workers) banned AI-generated change descriptions from his team: "AI was writing change descriptions that were worse than useless to me as I tried to review PRs: outlining details of the code that could easily be seen by looking at the code, but omitting the higher-level framing needed to understand broadly what the code is doing." Simon Willison surfaced and endorsed the take.

A sharp counter to the reflex of automating every artifact: the PR description is a human-to-human interface, and its whole job is the why — the framing a model can't infer from the diff and the reviewer most needs. The rule for your own tooling: let agents draft the mechanical stuff (changelog lines, test scaffolds), but keep intent-carrying prose — PR framing, advice rationale, client-facing "here's what changed and why" — human-authored or human-rewritten. Pairs directly with the Bun item: a reviewer who's told to break the code still needs a human to explain what it was for.

The architecture is the story: it's full-duplex — it can listen and speak simultaneously, backchannel with "mhmm," and let you talk over it — and for anything needing web search or real reasoning it delegates to GPT-5.5 in the background and folds the result back into the conversation while it keeps the flow going. The pattern to steal isn't the voice, it's the two-tier "conversational front / reasoning back" split: a fast, cheap, always-responsive model owns the interaction loop and dispatches to an expensive frontier model only for the queries that need it — the same price/performance routing this desk has been tracking, applied to latency rather than cost. For any real-time surface (a voice adviser intake, a live chat co-pilot) this is the shape: never make the user wait on the smart model; keep a fast one talking and stream the deep answer in when it lands.

Build Patterns

One primitive worth internalising: capture the deterministic part of an agent's work once, verify it, and never pay the model to redo it.

Auto: compile the deterministic slices of an agent run into verified WASM — "nothing is figured out twice"GitHub · 8 Jul

An open-source "AGI compiler" that records agent behaviour at runtime, extracts the parts that are secretly deterministic, and compiles them to sandboxed WebAssembly. It runs tiered: a fast compiled path for familiar inputs, a frontier-model fallback for novel ones — which then triggers recompilation, so the system ratchets. Reported result on a test stream: marginal cost dropped from ~59 µ$ to ~2 µ$ per task (6.4x cheaper) at steady state. Compiled artifacts are capability-confined (zero WASM imports; the loader refuses anything else), and each ships a manifest of measured eval scores, cost bounds and latency — not estimates. Runtime guarding uses conformal prediction to decide compiled-vs-model per input.

The idea is bigger than the repo: most "agent" workflows are mostly deterministic with a few genuinely fuzzy steps — so paying a frontier model to re-derive the deterministic 90% every run is pure waste. Capture it once, verify it against a contract, and serve it from cheap sandboxed code with the model as fallback. That's the cost thread and the verification thread fused: you get cheaper, faster, and auditable at once, because the hot path is now inspectable code with a capability boundary instead of an opaque model call. Worth a spike on any high-volume, mostly-repetitive agent loop you're running today.

MCP spec 2026-07-28 final lands in ~3 weeks — a breaking, stateless rewrite of the transportProtocol · deadline 28 Jul

The Model Context Protocol's next spec (release candidate out, final on 28 July) is its largest revision since launch and breaks from 2025-11-25. Headline changes: a stateless core — the initialize handshake and Mcp-Session-Id header are gone, so any request can hit any server instance behind a plain round-robin load balancer; MCP Apps, letting servers ship interactive HTML rendered in a sandboxed iframe; Tasks graduating to an official extension for long-running work (tasks/get, tasks/update, tasks/cancel); and OAuth/OIDC-aligned auth with mandatory iss validation.

If you run or consume MCP servers, this is a real migration, not a version bump — transport code and auth both change. The upside is significant: stateless routing kills sticky-session infrastructure, and the Tasks extension gives you a standard way to model the long-running, approval-gated agent work the ships above all rely on. Put the 28 July cutover on the calendar now and audit which of your servers carry session state today.
