The agent's surface is the product

Two of today's sharpest signals aren't about a model at all — they're about where you meet the agent. Julian Storer, who spent 30 years building JUCE and the audio tools half the industry ships on, says the terminal is the wrong surface for a coding agent and open-sourced Juggler to prove it: sessions as branching documents, context you can edit, every tool call inspectable. It's a thesis about steering and observability wearing the clothes of a product launch. Underneath, Claude Code's own mid-July releases quietly added hard caps on subagent spawns and web searches — the plumbing that stops an agent surface from running away with your bill. And OpenMarkdown answers a narrower version of the same question: how do a human and an agent write to the same file at once without clobbering each other? The model layer made its own noise too — Anthropic reversed course and will keep Fable 5 in paid plans from tomorrow — but Simon Willison's read is that this was never about generosity: it's what happens when a rival's flagship makes "no best model" untenable.

Julian Storer (creator of JUCE): the terminal is the wrong surface for a coding agentjuggler.studio · 18 Jul

Storer — 30+ years on JUCE, the Tracktion DAW and the Cmajor DSP language — makes a blunt case with Juggler: "I find a terminal UX horrible for all those tasks." Agent work is heavy text-editing and information-scanning, and a scrolling log is a bad container for it. His alternative treats a session as a document, not a transcript — any point can branch into a recursive sub-thread you navigate, duplicate, backtrack or undo; tool calls, approvals and the raw JSON context are all visible and editable in place; and the core loop (read/write/bash, context items, slash commands) is JavaScript plugins you can fork or replace.

This is really an argument about steering and observability, and it's worth taking seriously even if you never touch Juggler. The hard problems in agent products aren't "can it call the tool" — they're "can I see what it saw, correct the context, and rewind one bad step without nuking the run." If your product exposes an agent to users, ask where your hidden state lives: branchable history, editable context, and inspectable tool calls are becoming table stakes, and a chat log hides exactly the things users need to intervene on.

The product behind Storer's argument. Juggler runs locally, remotely, or both; conversations are branchable documents synced across a desktop and browser client at once; and its entire toolchain — context providers, LLM-loop strategies, slash commands, even read/write/bash — is a JavaScript plugin surface you can inspect and swap. Two takeaways. First, a reference implementation of the observable-agent pattern you can read rather than reverse-engineer — the plugin boundaries alone are a good map of what an agent harness actually needs. Second, a market read: the CLI won the first round of agent UX because it was fast to build, not because it was right. If a serious toolmaker is betting the GUI comes back for agent steering, that's a cue for anyone designing an agent-facing product surface.

Claude Code (v2.1.212–215): background /fork, and hard caps on runaway agentscode.claude.com · 17–19 Jul

Three days of releases with a consistent theme — making autonomy safe by default. /fork now copies a conversation into its own background session (the in-session spawner becomes /subtask); there's now a session-wide cap on WebSearch calls (default 200, CLAUDE_CODE_MAX_WEB_SEARCHES_PER_SESSION) and a matching cap on subagent spawns (default 200, reset by /clear); MCP calls over 2 minutes auto-background; and as of 19 Jul, /verify and /code-review no longer fire automatically — you invoke them.

The caps are the tell: as agents spawn agents, the failure mode shifts from "wrong answer" to unbounded fan-out — a loop that quietly burns 200 searches or 50 subagents. If you're building anything that lets an LLM spawn work, copy the pattern verbatim: a hard per-session budget on the expensive verbs (search, sub-spawn, tool time), tunable and resettable. The pull-back on auto-running /verify is the same instinct — expensive side effects should be opt-in, not ambient.

Build Patterns

One clean answer to a problem everyone building co-pilots hits: shared writes.

OpenMarkdown: human + agent co-edit one file, with optimistic concurrencyopenmarkdown.dev · 18 Jul

A local-first Markdown editor (Tauri 2, no telemetry, no account) whose whole point is the shared-write model: your agent reads, writes and co-edits the same .md file you're in, reachable over CLI, an agent plugin, or MCP. You hand it a task inline with - [ ] [@agent] task and it edits live alongside you. The mechanism worth stealing: writes use hash-based optimistic concurrency — if a section changed since the agent last read it, the agent's write is rejected and it's forced to re-read, rather than silently overwriting your edit.

Every co-pilot that writes into a document, canvas or record hits this race, and most hand-wave it. Optimistic concurrency with a per-section hash is the right default: no locks (which stall the human), no last-writer-wins (which eats edits) — a stale agent write just triggers a re-read against fresh state. If your product has an agent mutating shared, human-visible state, design the conflict path first; a rejected-and-retry write is far safer than a confident clobber, and it's a handful of lines, not a CRDT.
