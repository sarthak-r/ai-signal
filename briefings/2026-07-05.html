Two directly copyable practices: dedicated cheap review-subagents, and cross-model auditing as a QA gate — don't let the model that wrote the code be the only one that reviews it. Budget the loop by the task, not the call, and always have a different model mark the homework.

Ronacher: "Better Models, Worse Tools" — Opus 4.8 & Sonnet 5 regress on custom edit schemas because they're overfit to Claude Code's edit toolArmin Ronacher · 4 Jul

A counterintuitive, actionable regression. When Ronacher points the newest Claude models at Pi's nested edits[] edit-tool schema, they get oldText/newText right but fabricate fields that don't exist — requireUnique, oldText2, type, id, matchCase, forceMatchCount and more. Older models (including Opus 4.5) handled the same schema fine. His diagnosis: RL training on Claude Code's flat file_path/old_string/new_string edit tool gave the newer models a strong prior that "fights you harder" on a differently-shaped third-party schema. The fix that worked: Anthropic's strict mode (constrained decoding), which permits only valid keys.

A blunt warning for anyone building a coding harness or any custom tool contract: SOTA is not monotonically better at your schema. Upgrading the model can quietly break tool-calling you'd tuned on the previous version. Two takeaways: turn on strict / structured-output enforcement for every tool, and add a per-model tool-calling regression test to your eval suite — "the model got smarter" is exactly when to re-check the plumbing. Ronacher's open question — should harnesses ship model-matched tool variants? — is worth watching.

Claude Code Agent Teams: teammates that message each other and argue toward the answer — "scientific debate," productizedClaude Code · v2.1.199

Now documented and still being refined this week (through v2.1.199), Agent Teams is the sharpest departure from subagents: teammates are full independent sessions that message each other directly via a mailbox and self-claim from a shared, file-locked task list — no main agent as intermediary. The reference pattern is adversarial: "Spawn 5 teammates to investigate different hypotheses. Have them talk to each other to try to disprove each other's theories, like a scientific debate. Update the findings doc with whatever consensus emerges." The docs are candid about cost — each teammate is a separate context window, tokens scale linearly, start with 3–5. Enable with CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1.

This is Willison's cross-model audit and Litt's "stay a participant" turned into a UX primitive: the answer that survives teammates actively trying to refute each other is far likelier to be the real root cause than one agent's first plausible theory. The anti-anchoring structure is the point. For a builder, it's both a tool to adopt (parallel PR review by lens, competing-hypothesis debugging) and a pattern to copy in your own product — verification by adversarial multiplicity, not by trusting one pass.
