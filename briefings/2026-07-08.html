Ship Log

Three ships, one boundary: keep the model swappable, but own the parts around it — the memory, the file I/O, the proof of work.

Rowboat: a local-first coworker whose memory is a Markdown graph you can read, edit, or deleteShow HN · 6 Jul

An open-source (Apache-2.0) desktop alternative to Claude Desktop / Cowork that indexes your email, meeting notes and browsing into a knowledge graph stored as plain Markdown files, which every "work surface" (inbox, notes, browser, parallel coding) reads for context. It runs on any LLM, including local models via Ollama or LM Studio — no cloud memory silo. ~94 points on Show HN.

The interesting part isn't the app, it's the memory model: agent working memory as inspectable, hand-editable Markdown rather than an opaque vector store you can't audit. For any product where a human must be able to see and correct what the agent "knows" — a client file, an advice history — that's the right default, and it's forkable today. Steal the pattern even if you don't ship the app: make the agent's memory a document, not a database.

OfficeCLI: one binary that lets an agent read, edit and render Word/Excel/PowerPointGitHub · 6–7 Jul

An Apache-2.0, self-contained binary (embedded .NET, no Microsoft Office install) giving agents a command line over .docx, .xlsx and .pptx — 350+ Excel functions, a native OOXML pivot engine, and an HTML/PNG render mode that closes the render → look → fix loop. Document manipulation is a real gap in agent tooling, and this fills it deterministically — the agent edits a real spreadsheet and can screenshot the result to verify its own work, instead of hallucinating cell math. For a wealthtech build this is directly on-target: SoAs, fact-find spreadsheets and client letters are Office files, and a self-hosted binary means the documents never leave your machine. Spike it against whatever brittle python-docx glue you're maintaining.

Build Patterns

Two shapes worth stealing: make an agent's actions provable after the fact, and invoke the agent from the surface the team already lives in.

Halo: hash-chain every agent action into a tamper-evident log anyone can verifyShow HN · 7 Jul

~4,300 lines of Python (Apache-2.0) that wraps your agent entrypoint and records every tool call, model call, data access and approval as one entry in an append-only, hash-chained .jsonl log — delete a single line and the verification report catches it. The framing is sharp: give an agent the same prompt 50 times and you get 50 slightly different runs, so the only thing worth auditing is what actually happened at runtime. Cited hook: a 2026 survey where 88% of enterprises hit an agent security incident but only 21% had any runtime visibility.

This is the "verification becomes infrastructure" thread from last week made concrete and dependency-free — an audit trail you can hand a regulator that proves what the agent did without trusting the vendor who ran it. The pattern to internalise: for consequential agents, log at the action boundary (tool/model/data/approval), append-only, hash-chained — so "what did the agent do to this client's file?" has a cryptographic answer. Iress ships the tamer version of the same idea below (an audit trail on every CRM write); Halo is the primitive.

Ogment: "your AI coworker, in Slack — just tag @O" — the mention as agent invocationProduct Hunt · 7 Jul

A Slack-native agent (125 upvotes) invoked the way you already ping a colleague: @-mention it in the channel and it takes the action — no new app, no context switch, the agent lives where the conversation already is.

The pattern worth naming is invocation-in-place: the agent's entry point is a gesture the team already makes (a mention, a reply, a reaction), not a destination they have to visit. It's the same "graft onto an existing surface" move this desk flagged last week, applied to triggering the agent rather than just embedding it. For an adviser tool, the read is concrete — the agent should be summonable from inside the CRM thread or the email, not a separate portal someone has to remember exists.
