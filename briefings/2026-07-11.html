Cursor 3.11 already hooked tool execution and file/shell work; the new release adds hooks around the agent conversation itself — beforeSubmitPrompt, afterAgentResponse, afterAgentThought, subagentStart, stop, compaction, turn completion. The stated purpose: "observe output and reasoning, control subagents, and build self-correcting loops with cloud agents." Shipping alongside: side chats (/side, /btw) that branch off the main thread with shared context and can be re-merged, searchable agent transcripts (Cmd+K), and cloud agents that can drive their own computer to produce a demo artifact of their work.

This is the verification-as-infrastructure thread turned into an API. Until now, gating an agent meant checking its output — a diff, a commit, a completion flag. Hooks on afterAgentThought and subagentStart let you inspect and interrupt the reasoning and the fan-out as they happen: kill a subagent that's off-spec, inject a check before a prompt is submitted, halt on a policy trip. That's the machine-checked contract Mollick's spec needs, wired into the loop. If you run coding agents at any volume, the build to prioritise is the hook layer that enforces your spec mid-run — not another model wrapper. And note the pattern generalises: any agent product you ship should expose the equivalent observe-and-steer surface, because that's what a reviewer (or a regulator) will demand.

FARMA: agents can be attacked through their remembered reasoning, not just their factsarXiv · 6 Jul

New work (arXiv:2607.05029) on forged-reasoning attacks against LLM agent memory: instead of poisoning stored facts, the attack poisons the agent's remembered reasoning traces, so a future run retrieves and trusts a manipulated line of thought — plus a set of defenses to test against. It pairs with recent memory-sycophancy benchmarks probing when an agent should reject its own stored "memory" as evidence.

If you're shipping any agent with persistent memory — and an advice assistant that remembers a client across sessions is exactly that — this is a threat model you have to design against before launch. The uncomfortable implication: your memory store is now an attack surface with the authority of the model's own past reasoning, and naive "just remember everything" architectures inherit that risk silently. Add memory provenance and a scope/trust check to your retrieval path, and treat stored reasoning as untrusted input, not gospel.
