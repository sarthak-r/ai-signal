The bill you're not counting

Agentic speed is effectively free now — and this week's signal is the hidden bills that come attached. Armin Ronacher names the deepest one: agents strip out the friction that used to force teams to coordinate, so people keep shipping long after their shared understanding of the system has quietly collapsed — "the tower does not fall, and so we do not notice what was lost." The token receipts make the invisible visible: a viral teardown shows Claude Code burns ~33k tokens before it even reads your prompt (4.7× OpenCode's 7k), while Ploy's production swap to GPT-5.6 came in 2.2× faster, 27% cheaper, half the output tokens — but only after untangling provider-specific tool-call quirks. The corollary ships are disciplines, not generators: Perplexity's Computer grows a self-improving memory graph, Claude Code adds auto-mode guardrails against destructive commands, and GitHub Dependabot now waits three days before trusting a new release. Speed is cheap; the judgment around it is the work.

The sharpest warning of the week: what agent velocity quietly costs that no dashboard shows.

Armin Ronacher: agents keep the tower rising even after the shared language holding it up has collapsedlucumr.pocoo.org · 13 Jul

Ronacher (Flask, Sentry) reaches for Bruegel's Tower of Babel, but inverts the moral. At Babel, lost shared language stopped construction; with coding agents, construction continues. His thesis: "The shared language of a software project is not English or Python but the common understanding of what its concepts mean" — and "agents remove much of that friction… none of us necessarily has to talk to the others." Code compiles, tests pass, PRs merge, and the conceptual coherence rots underneath, unnoticed because "the tower does not fall."

This is the exact downstream cost of the throughput spike Willison flagged last week — velocity up, coordination optional, decay invisible. For a small team letting agents fan out across a codebase, the failure mode isn't a broken build; it's five locally-correct changes that no longer share a mental model. The counter-investment is deliberate: enforced review that checks intent not just diffs, living architecture docs, and naming/concept discipline agents must conform to. Treat shared understanding as a resource agents spend, and budget for refilling it.

The pattern to steal for any client-facing tool: memory isn't a feature bolted on, it's a compounding asset that lowers marginal cost per task. For a wealthtech product, a per-client context graph — prior goals, constraints, decisions — is exactly the moat that makes the tenth interaction cheaper and sharper than the first. Note the flip side: a memory graph over client financials is a governance surface you must be able to inspect and purge.

Build Patterns

Two hard-numbers reports that turn "agents are expensive" from a feeling into a measurement.

Ploy migrates a production agent to GPT-5.6: 2.2× faster, 27% cheaper, half the output tokens — after fixing what portability actually breaksploy.ai · 14 Jul

Ploy's agent builds and edits live marketing sites — it plans a page, reads the repo, writes components, screenshots its own work and decides when it's done. Swapping Claude Opus 4.8 → GPT-5.6 Sol delivered 2.2× faster completion, 27% lower cost, ~half the output tokens, and tighter code (one matched page: 2,508-char CSS with 45 variables vs Opus's 17,957 chars / 174 variables). But the migration exposed provider-specific assumptions everywhere: GPT-5.6 sends all 25 tool parameters on every call, filling unused ones with plausible junk (offset: 0, siteId: "0000…"), which silently broke their file-reads until caught.

Two builder lessons. First, model migration is now a measurable ROI decision, not a rewrite you avoid — but only if you have evals to prove the swap actually improved cost/latency/quality on your tasks, not a benchmark's. Second, and more durable: your tool layer must be defensive against how each provider fills arguments. Validate and reject implausible tool-call params (a zeroed UUID, a default offset) rather than trusting them — that's the difference between a portable agent and one welded to a single vendor's quirks. The capability-overhang gap swyx keeps naming is only harvestable if you can switch models cheaply.

The tool-description tax: Claude Code sends ~33k tokens before your prompt; OpenCode sends ~7kHacker News · 13 Jul

A widely-shared teardown measured the cold-start overhead of coding agents. Asked only to "print OK," Claude Code sent ~32,800 tokens in the first request vs OpenCode's ~6,900 — a 4.7× gap. The bulk is tool descriptions (~24k tokens for 27 tools) plus a ~6.5k system prompt. With MCP servers and config loaded, the initial payload balloons to ~75k tokens — ~37% of a 200k context window consumed before reading a line of your code. Prompt caching cuts the dollar cost of resending, but not the context space those tokens permanently occupy.

This is the token version of Ronacher's invisible cost: overhead you're not looking at until it degrades performance. The build takeaway is tool and context hygiene as a first-class concern — every MCP server and tool you wire in taxes every turn's context budget and pushes the model toward the lossy middle of its window (the "context rot" effect). Audit what your agent loads at cold start; gate rarely-used tools behind lazy/scoped loading rather than mounting all of them globally. Fewer, sharper tools beat a kitchen sink — for cost and for accuracy.

Your Radar

A supply-chain default that quietly changes how fast new code reaches you.

GitHub Dependabot now waits 3 days before opening a version-update PR — a default cooldown against supply-chain attacksgithub.blog · 14 Jul

Dependabot version updates now default to a three-day cooldown: it holds off opening an update PR until a new release has sat on its registry for at least 72 hours. GitHub's reasoning is blunt — "new releases are a common entry point for supply chain attacks where a compromised or broken version can reach your dependency updates before maintainers and the community have caught it." Security updates still open immediately; the cooldown applies only to version bumps, and you can tune or disable it via the cooldown option in .github/dependabot.yml.

This is the same instinct as the day's other guardrails, applied to dependencies: a deliberate pause is cheap insurance against trusting something too fast. The pattern generalises to agent-driven pipelines — a fresh package (or a freshly-generated config an agent wants to install) is at its most dangerous in its first hours, before anyone's vetted it. If you let agents touch dependencies or CI, adopt the same posture: quarantine-then-trust, not trust-then-discover. For a regulated fintech stack, a mandatory soak window on new third-party code is the kind of control an auditor will thank you for.
